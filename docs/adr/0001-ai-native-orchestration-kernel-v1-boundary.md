# ADR 0001: AI-native orchestration kernel v1 boundary

## Status

Accepted

## Context

Multica already has a working Issue, Agent, Runtime, Skill, daemon, and Agent Task model. Agent Tasks are the executable unit claimed by runtimes through the daemon and tracked through the existing task queue lifecycle.

The AI-native orchestration kernel adds planning, coordination, evidence, and recovery above that execution layer. Without a clear v1 boundary, the kernel could drift into a replacement task queue, a new daemon protocol, a full DAG scheduler, or an LLM planner before the core state machine is correct.

## Decision

The v1 orchestration kernel is an Issue-scoped, server-owned decision layer above the existing Agent Task lifecycle.

- An Issue can have at most one active Orchestration Run.
- Workspaces without orchestration enabled keep the existing direct issue-to-task enqueue path.
- The initial schedule is a deterministic linear chain: plan, execute, verify.
- The plan node is generated by the server, not by an LLM or agent.
- Executable nodes dispatch existing Agent Tasks through a Runtime Adapter.
- The daemon protocol and Agent Task queue remain the execution boundary.
- Kernel Events and Node Evidence are persisted as the source of truth for audit, recovery, and observability.
- Node recovery is node-scoped; completed nodes, events, and evidence are preserved.
- Human approval is conditional, not mandatory for every run.
- The first UI surface is the Issue Detail Decision Panel, not a standalone orchestration page.
- The first API surface is issue-scoped read plus approval actions, not generic orchestration CRUD.

## Consequences

This keeps the first kernel implementation compatible with Multica's existing issue, agent, runtime, skill, task queue, daemon, and frontend state-management models.

The trade-off is that v1 intentionally does not support full parallel DAG scheduling, workspace-wide automatic skill planning, LLM-generated plans, a verifier agent by default, or a new runtime protocol. Those can be added later behind explicit extensions once the kernel state machine, evidence contract, recovery behavior, and observability model are stable.
